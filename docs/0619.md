 ## 1. 问题：决策 token 位置定义有误

`0612.md §9` 对 StarCoder2-7B 在全正确样本集（FSC set，376 条）上的逐层排名轨迹分析，将"决策 token"定义为 `ground_truth_form` 的**第一个 token**。统计发现，376 条样本中有 **335 条（89%）** 的 `ground_truth_form` 是带命名空间前缀的点号形式（如 `tf.random.categorical`、`torch.linalg.lstsq`、`F.interpolate`），其首 token 分布为：

| 命名空间前缀 | 样本数 |
|---|---:|
| `tf.` | 234 |
| `torch.` | 61 |
| `F.`（`torch.nn.functional`） | 30 |
| `sklearn.` | 5 |
| `tensorflow.` | 3 |
| `sns.` / `mixture.` | 各 1 |

也就是说，§9 中所追踪的"决策 token"在约 89% 的样本上其实是 `tf`/`torch`/`F` 这类**库导入别名**，而不是该库内的具体 API 名称（`random`/`linalg`/`interpolate` 等）。这与 `06_mechanism/src/lens_analysis.py` 中 `shared_api_decision_prefix` 的约定不一致——后者会先剥离 replacement/deprecated 两种形式的公共命名空间前缀，只在剩余部分上定义"决策位置"，§5.6 的 stable decision depth 也是基于这一约定计算的。§9 的实现遗漏了这一步，导致逐层轨迹分析的对象错位为"模型何时决定使用哪个库"，而非"模型何时决定调用库内的哪个 API"。

## 2. 修正方案

在 `src/correct_selection.py` 中新增 `split_at_namespace_dot`：将 `ground_truth_form` 在第一个 `.` 处切分为 `(namespace_prefix, decision_form)`：

- 含 `.` 的形式（335/376）：`namespace_prefix` 为第一个 `.` 及之前的部分（如 `"tf."`、`"torch."`），`decision_form` 为其余部分（如 `"random.categorical"`、`"linalg.lstsq"`）；
- 不含 `.` 的形式（41/376，如 `"GaussianMixture"`）：`namespace_prefix=""`，`decision_form` 与原值相同，不受影响。

`run_topk_trajectory.py` 中据此将 teacher-forcing 的前缀从 `decision_prefix` 改为 `decision_prefix + namespace_prefix`，待预测的 suffix 从 `ground_truth_form` 改为 `decision_form`，**决策 token** 取 `decision_form` 的第一个 token。其余流程（逐层 Logit Lens / Tuned Lens 投影、top-10 进入层、稳定决策深度、JSD-to-final、竞争 token 统计）不变。在原 376 条 FSC 样本上重新跑 Step 1 + Step 2，输出见 `output/starcoder2_7b_decision_after_dot/`。

## 3. 修正后的整体结果

| 指标 | Logit Lens（修正后 / 修正前） | Tuned Lens（修正后 / 修正前） |
|---|---|---|
| top-10 进入层（均值 / 中位 / P25–P75） | 19.2 / **20** / 18–21　／　19.1 / 18 / 18–20 | 3.4 / **0** / 0–1　／　20.1 / 19 / 18–21 |
| 稳定决策深度（均值 / 中位 / P25–P75） | 27.9 / **28** / 27–31　／　24.8 / 24 / 21–29 | 24.8 / **30** / 23–31　／　28.3 / 29 / 26–31 |
| 未达到稳定决策深度的样本数 | 17/376（4.5%）／ 32（8.5%） | 54/376（14.4%）／ 33（8.8%） |

![fig1_emergence_layer_histogram_after_dot.png](pic/fig1_emergence_layer_histogram_after_dot.png)

两点变化值得注意：

1. **Logit Lens 的整体量级变化不大**（top-10 进入层 19.1→19.2，稳定决策深度 24.8→27.9），因为 Logit Lens 在中间层的读出本就以 BPE 噪声为主，命名空间 token 与 API 名称 token 对它而言难度相近。
2. **Tuned Lens 发生了质变**：top-10 进入层均值从 20.1 骤降至 3.4，其中 **256/376（68%）的样本在第 0 层（首个 decoder 层输出）就已经把正确的 API 名称排进 top-10**，部分甚至直接排第 1（如 `interpolate`、`divide`）。这说明：一旦命名空间前缀（`torch.`/`tf.`）被确定，Tuned Lens 训练得到的 translator 在最浅层就已经能从残差流中读出"该命名空间下接下来最可能的 API 名称"——即库一旦确定，**API 名称本身的可预测性远高于库的选择**。这一现象的稳健性（是否依赖 Tuned Lens 训练数据中类似上下文的覆盖度）需要进一步检验，本节先如实报告该数值。

## 5. API 文本是否已出现在上文中的对比

对每条样本检测正确 API 的文本形式（含别名）是否已出现在 prompt 上文中，分为两组：**API 已在上文中出现**（27 条，7.2%）与 **API 未在上文中出现**（349 条，92.8%）。

| 分组 | top-10 进入层（Logit / Tuned） | 稳定决策深度（Logit / Tuned） |
|---|---|---|
| API 已在上文中出现（27） | 20.2 / 9.8 | 26.7 / 23.7 |
| API 未在上文中出现（349） | 19.1 / 2.9 | 28.0 / 24.9 |

![fig4_api_in_context_after_dot.png](pic/fig4_api_in_context_after_dot.png)

API 已在上文中出现的样本，其 Tuned Lens top-10 进入层（9.8）反而**远晚于**未出现的样本（2.9）。结合 case study 可以看到：当上文已出现*同名但跨库*的 API 调用（例如上文写过 `tf.linalg.lstsq`，目标是 `torch.linalg.lstsq`）时，该名称会成为本库内的强竞争候选，反而**延迟**了正确 API 名称进入 top-10 的时机。92.8% 的样本属于"未出现"一类，说明这批 API 的正确预测主要依赖模型的参数化知识，而非对上文文本的直接复制。

## 6. Top-10 候选集合中的竞争 token 构成

统计各样本在 top-10 进入层之前，各层 top-10 中除正确 token 外的 token 类型分布：

| | Logit Lens | Tuned Lens |
|---|---|---|
| 标识符片段占比 | 85% | 94% |
| 高频候选 token | `constant`(265)、`contrib`(237)、`Variable`(105) | `sparse`(84)、`categorical`(74)、`cast`(74)、`math`(74)、`divide`(71)、`solve`(42) |

![fig3_competitor_profile_after_dot.png](pic/fig3_competitor_profile_after_dot.png)

两种 Lens 下的高频竞争 token 都是**同一命名空间下的其他 API 名称**：例如 `tf.constant`/`tf.contrib`/`tf.Variable` 与 `tf.random`/`tf.divide` 竞争，`cho`/`solve`（`torch.linalg.cholesky`/`torch.linalg.solve`）与 `linalg` 竞争。这表明模型在决定调用哪个具体 API 时，主要的竞争对象是**同一库内语义相近的其他 API**，而非语法噪声或其他库的命名空间。

## 7. case study：同一样本的修正前后对比

复用 §9.6 的三个样本，对比"决策 token"在修正前后的差异：

| 样本 | `ground_truth_form` | 修正前决策 token / top-10 进入层 / 稳定决策深度 | 修正后决策 token / top-10 进入层 / 稳定决策深度 |
|---|---|---|---|
| `qwencoder-3b-1389` | `tf.random.categorical` | `'tf'` / 18 / 25 | `'random'` / 16 / 31 |
| `codegemma-2b-357` | `torch.linalg.lstsq` | `'torch'` / 17 / 21 | `'linalg'` / 21 / 31 |
| `codegemma-2b-764` | `GaussianMixture` | `' Gaussian'` / 21 / 31 | `' Gaussian'` / 21 / 31（不含 `.`，不受影响） |

前两个样本修正后稳定决策深度都推迟到第 31 层（即接近输出层才完全稳定），说明"选择 `tf.` 还是 `torch.`"在这两个样本中其实比"选择 `random.categorical` 还是 `linalg.lstsq`"更早被模型确定下来——后者才是更晚收敛、更值得分析的决策。第三个样本 `ground_truth_form` 本身不含 `.`，修正前后完全一致，是 41 条不受影响样本之一。

## 8. 新增 case study（按 top-10 进入层分桶）

从修正后的三个分桶（top-10 进入层 < 18 / 18–21 / > 21）各选一个样本，均以 Logit Lens 为视角。

### 8.1 `codegemma-2b-864`（top-10 进入层 = 14，浅层组）

library: tensorflow，`ground_truth_api`: `tensorflow.random.categorical`，`namespace_prefix`: `"tf."`，决策 token: `'random'`。

| 层 | rank | JSD | top-3 候选 |
|---|---:|---:|---|
| 13 | 309 | 0.651 | `ocl`, `math`, `cast` |
| **14** | **9** | 0.644 | `math`, `Variable`, `ocl` |
| 15 | 2 | 0.635 | `constant`, `random`, `cast` |
| 16 | 1 | 0.513 | `random`, ` random`, `cast` |
| ... | | | |
| 30 | 1 | 0.166 | `random`, `cast`, `mult` |
| 31 | 1 | -0.000 | `random`, `squeeze`, `argmax` |

排名在层 13→14 间从 309 降至 9。层 16 即短暂达到 rank-1，但随后被 `argmax`/`cast` 反超（层 17 排名回落到 25），直到层 30 才重新回到并稳定在 rank-1（稳定决策深度 = 30）。**这是一个"提前进入 top-10 后又被同库竞争 API 反超、需在深层重新确立"的例子**，体现了"`tf.` 确定后，具体调用哪个 `tf.xxx` 函数"这一决策本身的不稳定性。

### 8.2 `codegemma-2b-170`（top-10 进入层 = 21，中层组）

library: pytorch，`ground_truth_api`: `torch.linalg.solve`，`namespace_prefix`: `"torch."`，决策 token: `'linalg'`。

| 层 | rank | JSD | top-3 候选 |
|---|---:|---:|---|
| 20 | 270 | 0.644 | `random`, `empty`, `hub` |
| **21** | **7** | 0.604 | `cho`, ` Cho`, ` solve` |
| 22–30 | 4–12 | 0.31–0.53 | `solve`, `cho`, `tri` 交替排前三 |
| 31 | 1 | -0.000 | `linalg`, `solve`, `tri` |

排名在层 20→21 间从 270 降至 7，但此后长达 9 层（22–30）`solve`/`cho`（`torch.linalg.cholesky`/`solve`）持续压制 `linalg`，直至最后一层才反超。**这是一个"目标 token 早早进入 top-10、但被同语义域的近义 API 长期压制"的例子**：`torch.linalg.solve` 与 `torch.linalg.cholesky` 都是求解线性方程组的相关 API，模型在很深的层仍在二者之间摇摆。

### 8.3 `qwencoder-3b-529`（top-10 进入层 = 24，深层组）

library: pytorch，`ground_truth_api`: `torch.linalg.svd`，`ground_truth_form`: `"svd"`（不含 `.`，因代码中以 `linalg.svd(...)` 形式书写，决策 token 即 `svd` 本身），prompt 上文中已出现完整的 `torch.linalg.svd`。

| 层 | rank | JSD | top-3 候选 |
|---|---:|---:|---|
| 23 | 54 | 0.502 | ` _`, ` super`, `_` |
| **24** | **3** | 0.345 | ` _`, ` None`, ` sv` |
| 25 | 1 | 0.142 | ` sv`, ` SV`, `SV` |
| 26 | 1 | 0.134 | ` sv`, ` SV`, `SV` |

尽管 `torch.linalg.svd` 已在上文完整出现过，正确 token 仍要到第 24 层才进入 top-10——印证了 §5 的发现：上文中出现过的 API 名称不会被模型"直接复制"，其进入 top-10 的时机甚至可能晚于均值（19.1）。第 23 层的 top-3 候选（`_`/`super`）与 `svd` 在语义上无关，说明该样本的决策几乎是在第 24 层突然完成的。

## 9. 与 SFT+DPO §5.6 的关联（更新）

§5.6 报告的 base model（StarCoder2）**stable decision depth**（Tuned Lens 下，replacement vs. deprecated API 偏好不再翻转的层）均值为 **24.271**。修正前，§9 误用 Logit Lens 下"决策 token 即命名空间别名"的稳定决策深度（24.8）与之对比，量级巧合接近但**度量对象并不一致**。

修正后，本节在**同样剥离命名空间前缀**的口径下，测得 base model 在 FSC set 上 Tuned Lens 的稳定决策深度均值为 **24.8**（中位数 30）——与 §5.6 的 24.271 量级一致，且度量对象（"剥离公共/命名空间前缀后，API 名称的决策何时稳定"）与 §5.6 的 stable decision depth 定义在概念上对齐。这为"SFT+DPO 将 stable decision depth 从约 24 层压缩至 0.97 层"提供了一个度量口径一致的 base model 参照基线。

需要注意：修正后 Tuned Lens 的 top-10 进入层均值仅为 3.4（68% 样本在第 0 层即进入 top-10），而稳定决策深度仍高达 24.8——即"候选范围"很早就收窄到包含正确答案，但"排名第一且不再变化"要晚得多。这与 §9.7 原先"top-10 进入层与稳定决策深度相差 5–10 层"的结论方向一致，但修正后两者的差距（约 21 层）远大于修正前（约 4–8 层），说明 SFT+DPO 压缩的"24 层收敛过程"主要发生在**候选已经包含正确答案、但尚未排到第一**这一阶段。

## 10. 关于 case study 样本 `row_id` 命名的说明

`0612.md §9.6` 与本文档 §7 中的 `row_id`（如 `qwencoder-3b-1389`、`codegemma-2b-357`）来自 EDAPIBench 数据集自身的样本标识（格式为 `{来源模型}-{规模}-{编号}`），表示该 prompt/reference 对最初是为评测 codegemma-2b / qwencoder-3b / deepseek 等模型构造的，**与本实验的前向传播模型无关**——本实验（`run_topk_trajectory.py --model-key starcoder2_7b`）对所有 376 条样本的 teacher-forcing 前向传播均使用 **StarCoder2-7B**。FSC set 中仅有 6/376 条样本的 `row_id` 带 `starcoder2` 前缀（来自 repair/consistency 来源），§8.3 的新增 case study `qwencoder-3b-529` 与之前一样属于"`row_id` 含其他模型名，但前向传播模型为 StarCoder2-7B"的情形，行为上无误，仅命名容易引起误解。

## 11. 小结

1. 修正后，约 89% 样本的决策 token 从库命名空间别名（`tf`/`torch`/`F`）变为该库内的具体 API 名称（`random`/`linalg`/`interpolate` 等），与 06_mechanism/SFT+DPO §5.6 的"决策位置"约定对齐。
2. Logit Lens 下整体量级变化不大；Tuned Lens 下 top-10 进入层从均值 20.1 骤降至 3.4（68% 样本第 0 层即进入 top-10），稳定决策深度均值 24.8，与 §5.6 的 24.271 量级一致。
3. 竞争 token 构成从"BPE 噪声 + 跨库命名空间别名"变为"同库内的近义 API 名称"（如 `solve`/`cho` 与 `linalg` 竞争），更直接地回答了"模型在决定调用哪个 API 时与谁竞争"这一问题。
4. "API 文本已在上文出现"不再表现为"更早进入 top-10"，反而因引入跨库同名竞争而**延迟**收敛。
5. `06_mechanism`（OV 电路 / SFT+DPO LoRA 等实验，详见 `0605.md`）使用的是 `lens_analysis.py` 中已经剥离命名空间前缀的 `shared_api_decision_prefix` 约定，不受本次修正影响。
