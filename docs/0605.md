## 6. OV 电路在 linalg API 迁移案例上的精细化分析

### 6.1 实验动机

§5 中的 OV 电路分析以三个 PyTorch API 迁移 case 为研究对象（`torch.svd → torch.linalg.svd`、`torch.qr → torch.linalg.qr`、`F.upsample → F.interpolate`），发现 SFT+DPO 对最终层头的 OV 贡献改变最大。本节对两个 linalg 类 case（svd 和 qr，合称"linalg case"）做更精细的因果分析，并将跨模型一致性作为主要验证维度。选取 linalg case 的原因：base model 对这两个 case 的 logit_diff 均为**负值**（偏好弃用 token），SFT+DPO 需要做完全的方向逆转，信号对比度高于 `F.upsample → F.interpolate`（该 case 的 base model 已偏好替代 token）。

### 6.2 Direct Logit Attribution（DLA）结果：StarCoder2-7B

对 StarCoder2-7B 进行逐头 OV DLA，量化每个注意力头对 $\Delta_{\text{logit}} = \text{logit}(\text{rep}) - \text{logit}(\text{dep})$ 的贡献量及其在 base vs SFT+DPO 之间的变化。

| API 迁移 case | base $\Delta_{\text{logit}}$ | SFT+DPO $\Delta_{\text{logit}}$ | #1 贡献头（层/头） | flip_delta |
|---|---:|---:|---|---:|
| `torch.svd → torch.linalg.svd` | −2.06 | +35.03 | L31H2 | +1.83 |
| `torch.qr → torch.linalg.qr` | −3.16 | +32.75 | L31H2 | +1.69 |
| `F.upsample → F.interpolate` | +0.47 | +13.14 | L31H18 | +0.30 |

**linalg case 的 top-5 贡献头（svd 和 qr 共同出现的头）**：

| 层/头 | svd flip_delta | qr flip_delta | 备注 |
|---|---:|---:|---|
| L31H2 | +1.833 | +1.688 | 两个 case 均为 #1，贡献压倒性 |
| L31H1 | +1.510 | +1.518 | 两个 case 均为 #2 |
| L25H33 | +0.337 | +0.389 | 两个 case 共现 |
| L30H11 | +0.317 | — | svd only |
| L18H26 | — | +0.664 | qr only，第 3 大 |

两个 linalg case 的 top-2 头完全相同（L31H2、L31H1），且 flip_delta 量级高度接近（1.51–1.83 vs 其他头的 0.30–0.66）——这支持一个结论：**SFT+DPO 对 linalg 类迁移的改写集中于同一组 OV 头**，该机制具有 case 间的共性而非 case 特异性。

`F.upsample → F.interpolate` 的 #1 贡献头为 L31H18，flip_delta 仅 0.30（linalg case 的 1/6），与 linalg case 使用不同的头，且绝对贡献量小一个数量级——这说明两类迁移（命名空间前缀替换 vs 函数名替换）通过**不同的 OV 头集合**实现。

### 6.3 因果翻转分析（OV Flip Analysis）

Flip 分析在每个头上独立执行 OV 消融（将该头的 OV 输出置零），统计消融后 $\Delta_{\text{logit}}$ 的降幅（flip_score = rep_rank 改善幅度 + dep_rank 降幅），以定位因果贡献头。linalg case 的结果（StarCoder2-7B，SFT+DPO 版本）：

| case | #1 因果头（层/头） | flip_score | rep_gain | dep_drop |
|---|---|---:|---:|---:|
| `svd → linalg.svd` | L21H8 | 4510 | 3965 | 545 |
| `qr → linalg.qr` | L18H10 | 4548 | 4296 | 252 |
| `symeig → linalg.eigh` | L17H24 | 4684 | 4152 | 532 |
| `F.upsample → F.interpolate` | L29H30 | 1897 | 1076 | 821 |

关键观察：

1. **DLA vs Flip 分析不一致**：DLA 显示 L31H2 是 SFT+DPO 改变最大的头（直接贡献量最大），但 flip 分析的因果性最强头在层 17–21 附近（L21H8、L18H10、L17H24），且 flip_score（4500+）远高于 upsample 的最强头（1897）。
2. **深层头 vs 浅中层头**：L31H2 的大 flip_delta 来自 SFT+DPO 对其权重的直接修改（DLA 意义），而层 17–21 的因果头可能是 base model 就已建立的信息路由，SFT+DPO 通过上游改变向这些头输送了更多的 linalg 方向信号。
3. **linalg case 的因果性普遍强于 upsample**：flip_score 量级（4500 vs 1900）与 DLA 量级（|Δ|≈33–35 vs 13）方向一致，两种度量共同指向 linalg 迁移在因果强度上远大于 upsample 迁移。

### 6.4 跨模型一致性

对 StarCoder2-3B、StarCoder2-15B、DeepSeek-Coder-6.7B（SFT+DPO fine-tuned）进行相同的 OV DLA，验证机制是否具有模型泛化性：

| 模型 | `svd→linalg.svd`（base→SFT+DPO） | `qr→linalg.qr`（base→SFT+DPO） | `upsample→interp`（base→SFT+DPO） | 顶层贡献层 |
|---|---|---|---|---|
| StarCoder2-3B | −3.22 → **+33.25** | −3.13 → **+31.19** | +1.46 → +15.54 | L27 |
| StarCoder2-7B | −2.06 → **+35.03** | −3.16 → **+32.75** | +0.47 → +13.14 | L31 |
| StarCoder2-15B | −5.52 → **+36.31** | −4.94 → **+32.81** | +1.07 → +10.69 | L39 |
| DeepSeek-6.7B | −6.77 → **+39.28** | −0.27 → **+39.63** | +5.81 → +10.98 | L31 |

所有 4 个模型在两个 linalg case 上均呈现**完全方向逆转**（负 → 正 30+），量级高度一致（31–39），且最高贡献层均为各自架构的最后几层（层深度约 84–100%）。这说明 SFT+DPO 的 OV 电路改写机制具有**跨模型泛化性**，不依赖于特定的模型尺寸或架构细节。

DeepSeek-6.7B 在 qr case 上 base_logit_diff 仅为 −0.27（几乎中性），SFT+DPO 后达到 +39.63，改变幅度（39.9）高于 StarCoder2 系列（35–36），说明 DeepSeek 的 SFT+DPO 在 linalg 方向上的 OV 注入更为集中。

## 7. Prefix Tuning 作为 SFT+DPO LoRA 的替代方案

### 7.1 动机与设计

SFT+DPO 的当前实现采用 LoRA 适配器（rank=16，覆盖 q/v 投影）。本实验探索以 **Prefix Tuning** 替代 LoRA 作为参数高效微调方案，考察：
- Prefix Tuning 能否在相近参数量下达到与 SFT+DPO LoRA 接近的版本一致性行为？
- 两种方法的 OV 电路修改模式是否一致？

**Prefix Tuning 设计**：在每层的 KV 缓存前拼接长度为 $m$ 的可训练 prefix token 序列（共 $m \times n_{\text{layers}} \times n_{\text{KV}} \times d_{\text{head}}$ 参数）。以 SFT+DPO 同款 DPO 目标训练（reference policy = same backbone with prefix disabled）。

测试了四个变体（StarCoder2-7B，32层，4 KV 头，128 d_head）：

| 变体 | prefix 长度 $m$ | 可训练参数量 | 是否含 DPO anchor |
|---|---:|---:|---|
| cerpt_m16 | 16 | 524,288 | ✓ |
| cerpt_m64 | 64 | 2,097,152 | ✓ |
| cerpt_m256 | 256 | 8,388,608 | ✓ |
| pt_only_m64 | 64 | 2,097,152 | ✗（纯 Prefix Tuning） |

训练集：mixed_sft_train（2408 样本，1970 DPO pairs）；评估集：mixed_sft_test（285 样本）。

### 7.2 行为评估结果

| 变体 | deprecated_usage_rate ↓ | replacement_hit_rate ↑ |
|---|---:|---:|
| Base（无适配） | — | — |
| **cerpt_m16** | 5.26% | 11.93% |
| **cerpt_m64** | 1.40% | **15.09%** |
| **cerpt_m256** | 1.05% | **15.09%** |
| **pt_only_m64** | 2.11% | 13.68% |

从 m16 到 m64，replacement_hit_rate 从 11.93% 提升至 15.09%（+3.2 pp），deprecated_usage_rate 从 5.26% 降至 1.40%；m256 与 m64 的效果几乎**完全相同**（15.09% vs 15.09%），说明参数量在 m=64（2.1M）时已饱和，继续扩大 prefix 长度无收益。

去掉 DPO anchor 的 pt_only_m64 表现略差（13.68% vs 15.09%），说明 DPO 损失相比纯 SFT 目标对版本偏好有额外贡献，但贡献量不大（+1.4 pp）。

### 7.3 OV 电路对比

对上述四个变体在相同三个 API case 上运行 OV DLA，量化其 $\Delta_{\text{logit}}$：

| 变体 | `svd→linalg` | `qr→linalg` | `upsample→interp` |
|---|---:|---:|---:|
| Base | −2.125 | −3.250 | +0.438 |
| **SFT+DPO (LoRA)** | **+35.0** | **+32.5** | **+13.0** |
| cerpt_m16 | +0.5 | +0.125 | +1.375 |
| cerpt_m64 | +2.125 | +3.0 | +2.0 |
| cerpt_m256 | +2.125 | +4.563 | +3.688 |
| pt_only_m64 | +1.625 | +2.5 | +1.938 |

**结论**：Prefix Tuning 对 OV 电路的修改量级与 SFT+DPO LoRA 之间存在**数量级差距**（2–4.6 vs 32–35）。即使 m=256（8.4M 参数，远多于 LoRA 的 \~1.5M），linalg case 的 $\Delta_{\text{logit}}$ 也仅为 SFT+DPO 的约 1/8–1/13。

这说明：Prefix Tuning 通过 **attention pattern 调制**实现版本偏好注入（让模型在 KV 上下文中更多关注替代 API 的特征），而不是通过 OV 电路的输出值修改。这两种机制在最终行为上均能降低 deprecated_usage_rate，但 Prefix Tuning 对核心 OV 电路的影响极小，说明它通过更上游（attention score 分布）而非更下游（token logit 直接贡献）的路径实现改写。

## 8. 分库训练 SFT+DPO LoRA 的横向对比

### 8.1 实验设置

标准 SFT+DPO 在混合库训练集（mixed_sft_train，覆盖 numpy / pytorch / scipy / sklearn / seaborn / tensorflow 六个库）上联合训练单个 LoRA。本实验改为**每个库单独训练一个 LoRA**，与 base model 对比，考察：

- 各库是否受益程度不同？
- 哪些库通过专库 LoRA 能达到高 replacement_hit_rate？
- 哪些库对 SFT+DPO 训练不敏感？

**训练设置**：7 个模型 × 6 个库 = 42 个 LoRA；每个 LoRA 只在该库的 (deprecation, replacement) pair 数据上训练；评估在该库的 test split 上进行。

### 8.2 各模型 × 各库 结果

#### 8.2.1 PyTorch（最高改善）

| 模型 | base rep_hit | LoRA rep_hit | dep_rate (base → LoRA) |
|---|---:|---:|---|
| starcoder2_3b | 12.5% | **80.8%** | 20.0% → 0% |
| starcoder2_7b | 7.5% | **88.3%** | 25.0% → 0% |
| starcoder2_15b | 12.5% | **75.8%** | 23.3% → 0% |
| qwen2_5_coder_3b | 15.8% | **75.0%** | 18.3% → 0% |
| qwen2_5_coder_7b | 28.3% | **70.0%** | 14.2% → 0% |
| qwen2_5_coder_14b | 32.5% | **74.2%** | 9.2% → 0% |
| deepseek_coder_6_7b | 30.0% | **72.5%** | 14.2% → 0% |

所有 7 个模型在 pytorch 上的 LoRA 都将 deprecated_usage_rate 降至 **0%**，replacement_hit_rate 提升至 70–88%。PyTorch 是各库中改善最显著、最一致的库。

#### 8.2.2 TensorFlow（高改善）

| 模型 | base rep_hit | LoRA rep_hit | dep_rate (base → LoRA) |
|---|---:|---:|---|
| starcoder2_3b | 19.7% | **81.6%** | 0% → 1.3% |
| starcoder2_7b | 28.9% | **84.2%** | 2.6% → 1.3% |
| starcoder2_15b | 23.7% | **78.9%** | 2.6% → 2.6% |
| qwen2_5_coder_3b | 34.2% | **85.5%** | 1.3% → 1.3% |
| qwen2_5_coder_7b | 31.6% | **82.9%** | 0% → 1.3% |
| qwen2_5_coder_14b | 34.2% | **89.5%** | 6.6% → 1.3% |
| deepseek_coder_6_7b | 28.9% | **82.9%** | 2.6% → 1.3% |

TensorFlow 的改善同样显著（rep_hit: 20–34% → 79–89%）。值得注意的是部分模型 LoRA 后 deprecated_usage_rate 略有**上升**（starcoder2_3b: 0% → 1.3%），说明专库 LoRA 训练可能在某些 case 上引入小幅回归。

#### 8.2.3 NumPy（中等改善）

| 模型 | base rep_hit | LoRA rep_hit |
|---|---:|---:|
| starcoder2_3b | 2.6% | 10.5% |
| starcoder2_7b | 2.6% | 23.7% |
| starcoder2_15b | 7.9% | 34.2% |
| qwen2_5_coder_3b | 10.5% | 28.9% |
| qwen2_5_coder_7b | 13.2% | 21.1% |
| qwen2_5_coder_14b | 18.4% | **42.1%** |
| deepseek_coder_6_7b | 15.8% | 18.4% |

NumPy 的改善中等（2–18% → 10–42%），且在各模型间差异较大。qwen2_5_coder_14b 改善幅度最大（+23.7 pp）；deepseek_coder_6_7b 几乎无改善（+2.6 pp）。

#### 8.2.4 Scipy / Sklearn / Seaborn（弱改善或无改善）

| 库 | 典型结果 | 规律 |
|---|---|---|
| scipy | base: 5–20%，LoRA: 10–25% | 小幅提升（+0–5 pp），部分模型 dep_rate 反而微升 |
| sklearn | base: 0–20%，LoRA: 0–20% | **几乎无改善**，多数模型改善为 0 pp |
| seaborn | base: 0–21%，LoRA: 0–29% | 极少改善（样本量极小，结论可靠性低） |

sklearn 是各库中改善最差的库：7 个模型中，4 个 LoRA 与 base 的 replacement_hit_rate 完全相同（starcoder2_3b: 0%→0%，starcoder2_15b: 0%→0%，qwen2_5_coder_14b: 6.7%→6.7%，deepseek 6.7b: 20%→20%）。这可能与 sklearn API 的迁移模式有关：sklearn 的 deprecated 形式与 replacement 形式往往共享长前缀（如 `sklearn.linear_model.LinearRegression` 等），模型难以通过单 token 级别的偏好注入来区分。

### 8.3 跨库 × 跨模型汇总

| 库 | 7 模型平均 rep_hit 提升 | dep_rate 降至 0% 的模型数 |
|---|---:|---:|
| pytorch | +57.9 pp | 7 / 7 |
| tensorflow | +54.2 pp | 0 / 7（但均 ≤ 2.6%） |
| numpy | +13.7 pp | — |
| scipy | +3.9 pp | — |
| sklearn | +0.7 pp | — |
| seaborn | +2.6 pp（样本量小） | — |

**pytorch 和 tensorflow 的改善幅度（+54–58 pp）是 numpy 的 4 倍、sklearn 的 80 倍。** 这一两极分化的规律在全部 7 个模型上高度一致，说明是库本身的特性（而非特定模型的记忆能力差异）决定了 SFT+DPO 专库训练的效果上限。

一个合理假设是：pytorch / tensorflow 的迁移主要涉及**模块前缀替换**（`torch.` / `torch.linalg.` / `tf.compat.v1.` → `tf.`），决策 token 是命名空间前缀，模型对这类替换具有更高的可训练性（对应 §6 中 linalg case 的高 flip_score）；而 sklearn 的迁移更多涉及**整个类名变化**，决策 token 是完整的类名 token，偏好注入效率低。
